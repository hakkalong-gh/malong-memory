## 🔧 飞书专用工具链修复经验（2026-03-28 晚）

**事件**: DL 要求直接读取飞书 wiki / docx 页面，一开始我误判为“当前会话没有暴露 feishu_wiki / feishu_doc 工具”，后来排查并修复成功。

### 问题表现
- 一开始无法用飞书专用链路直接读取 wiki/docx
- 我错误地把“当前不可调用”理解成“工具链未恢复 / 不存在”
- 实际上是需要先确认 Gateway 侧插件注册与配置状态，而不是只凭当下调用印象判断

### 正确排查路径
1. 检查 `~/.openclaw/openclaw.json` 中 `channels.feishu` 配置
2. 确认是否显式启用了：
   ```json
   "tools": {
     "wiki": true,
     "doc": true
   }
   ```
3. 执行 `openclaw gateway restart`
4. 用 `openclaw gateway status` / `openclaw status` 验证：
   - Gateway 是否 running
   - 插件是否已注册：`feishu_wiki`、`feishu_doc`、`feishu_drive`、`feishu_bitable`
5. 再实际测试 wiki/docx 读取

### 本次修复结果
- 已在 `channels.feishu` 下补入：
  - `tools.wiki: true`
  - `tools.doc: true`
- Gateway 已恢复运行
- `feishu_wiki / feishu_doc` 已可正常使用
- 已成功：
  - 读取 wiki 页面
  - 读取 docx 页面
  - 将一个 docx 的内容复制到另一个 wiki 对应页面

### 重要教训
- **先验证，再判断**：不能因为一次调用失败，就说“工具不可用”
- **飞书专用工具链恢复的关键，不是浏览器，而是 `channels.feishu.tools` + Gateway 重启 + 状态验证**
- 以后遇到“明明装过技能但用不了”的情况，优先查：
  - 配置是否显式开启
  - Gateway 是否加载新配置
  - 插件是否在 `openclaw status` 中注册

### 可复用最短修复方案
```json
{
  "channels": {
    "feishu": {
      "tools": {
        "wiki": true,
        "doc": true
      }
    }
  }
}
```
然后执行：
```bash
openclaw gateway restart
openclaw status
```

### 后续补充：页面权限问题（同晚验证）

**新增结论**：
飞书专用工具链恢复后，**“能读不能写”不等于工具链又坏了**，很可能只是**目标页面对应 docx 没有写权限**。

**本次实测现象**：
- 页面 A：可读、可写，复制成功
- 页面 B：可读，但写入返回 `403`
- 在 DL 调整目标 wiki 页面权限后，再次测试：
  - 最小写入测试成功
  - 完整内容复制成功

**因此要区分两类问题**：
1. **工具链问题**：`feishu_wiki / feishu_doc` 不可调用、插件未注册、Gateway 未加载配置
2. **页面权限问题**：页面能读，但写入 `403`

**以后正确判断方式**：
- 如果 `feishu_wiki` / `feishu_doc` 能正常读页面，说明工具链大概率已恢复
- 如果写入单独报 `403`，优先判断为**目标页面权限问题**，不是工具链回退
- 先做一次**最小写入测试**，再决定是否让 DL 调整权限

**一句话经验**：
> 飞书链路恢复 ≠ 所有页面都可写；读得了但写不了，先查页面权限。
