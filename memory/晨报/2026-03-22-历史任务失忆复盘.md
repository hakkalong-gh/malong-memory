# 2026-03-22 历史任务失忆复盘

## 事件

今天在处理 IMA 笔记追加时，DL 明确提醒：
- 昨天已经安装过 ima-note skill
- 昨天已经成功在 IMA 旧笔记里追加过内容

但我第一反应仍然是：
- 重新找入口
- 重新判断是否有写入能力
- 让 DL 再提供链接/入口

这和今天上午忘记 Python 3.11 已安装，是同类错误。

---

## 根因

### 1. 读了记忆，但没有按任务触发检索
会话启动时读取了 memory 文件，但没有在真正动手前，针对 “IMA / 安装过的 skill / 昨天做过的操作” 进行定向 memory_search。

### 2. 依赖印象，不依赖证据
我下意识根据“当前没看到入口”做判断，而不是先验证：
- 以前是否成功过
- 有没有现成 doc_id / API / skill
- 有没有已经跑通的路径

### 3. 缺少“像做过的任务”的专项检查
对于这类明显像做过、像配过、像已经成功过的事情，没有先停下来问：
1. 这件事以前做过吗？
2. 有没有现成记录？
3. 我现在是基于证据还是基于印象？

---

## 新规则（强制）

### 规则 1：像做过的任务，先查记录再动手
凡是涉及以下关键词的任务：
- 安装
- 配置
- skill
- MCP
- API
- 笔记
- 昨天做过 / 上次做过 / 之前弄过

必须先：
1. memory_search 搜索历史
2. 检查 TOOLS.md / MEMORY.md / memory/*.md
3. 再决定是否重新安装、重新配置、重新找入口

### 规则 2：不准凭印象判断“有没有装过 / 配过 / 做过”
禁止直接说：
- 没装过
- 没配置
- 需要重新来
- 没入口

除非已经完成：
- 历史检索
- 当前验证
- 多源确认

### 规则 3：第一次卡住时，优先查历史成功路径
遇到任务受阻时，不只是换工具，也要先问：
- 历史上这件事是怎么做成的？
- 当时用的是哪个 skill / API / doc_id / 脚本？

---

## 执行清单

以后凡是“疑似做过”的任务，执行前先跑：

```text
[ ] 这事以前做过吗？
[ ] memory_search 了吗？
[ ] TOOLS.md / MEMORY.md / 当日 memory 查了吗？
[ ] 有没有现成 doc_id / 配置 / API / 脚本？
[ ] 当前结论是基于证据，不是印象吗？
```

---

## 标准工作顺序

1. 先查历史
2. 再做当前验证
3. 第一次异常就汇报
4. 汇报时明确说：
   - 我查了哪些历史
   - 我验证了哪些路径
   - 当前结论是什么

---

## 一句话教训

> 凡是“像以前做过”的任务，不准凭印象，先查记录，再动手。
